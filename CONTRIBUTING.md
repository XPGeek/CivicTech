# Contributing to DMV Water Watch

This guide is for anyone — human or agent — who wants to run, test, debug, or extend the app. If anything below is wrong, file an issue or a PR; the docs are part of the product.

> **Not a coder?** Read [`GETTING_STARTED.md`](./GETTING_STARTED.md) instead. It explains how to suggest sites, report incorrect grades, and help with outreach — no command line required.

> **One-line summary.** Clone, `npm install`, `npm run pipeline`, `npm run dev`. The map should render on http://localhost:3000.

---

## 1. Prerequisites

- **Node.js ≥ 20** (Node 23 is what the repo is developed against).
- **npm ≥ 10** (ships with Node 20+).
- Any modern browser. Chrome/Edge for PWA install testing; Safari for iOS testing.
- No accounts, API keys, or Mapbox tokens are required to run the local dev experience. The default map tiles are OpenStreetMap raster (token-free).

Optional for production deploys:

- Cloudflare account (free Pages + R2 tier).
- Mapbox token (if you want to switch from OSM raster to Mapbox Standard).

---

## 2. First-time setup

```bash
git clone <this-repo>
cd dmv-water-watch
npm install
cp .env.example .env.local         # optional; defaults work without changes
npm run pipeline                   # populate public/data/ with fresh artifacts
npm run dev                        # http://localhost:3000
```

The first `npm run pipeline` makes real API calls to USGS NWIS and NOAA. If you're offline, those connectors return empty and the build still completes — the fixture-backed Anacostia Riverkeeper and DOEE sondes connectors will still drive grades.

## 4. Common commands

| Command | What it does |
|---|---|
| `npm run dev` | Start Next.js dev server at http://localhost:3000. |
| `npm run pipeline` | Run all connectors → grade every site → write artifacts to `public/data/` AND `data/snapshots/dev/`. |
| `npm run connector:run -- <id>` | Run a single connector and print its emitted records to stdout. Useful for debugging a flaky source. |
| `npm run validate:sites` | Static check of `data/sites.json` — schema, bounding box, station references, DC-swim prohibition. |
| `npm test` | Run unit tests (Vitest). 49 connector tests + 17 grading tests. |
| `npm run test:watch` | Same, watch mode. |
| `npm run grading:test` | Only the grading-rubric tests. Fast feedback when you change `grading/v1.ts`. |
| `npm run typecheck` | `tsc --noEmit`. Run before opening a PR. |
| `npm run lint` | ESLint via `next lint`. |
| `npm run build` | Run the pipeline then build the Next.js static export to `out/`. |
| `npm run start` | Serve the static `out/` build locally. |

## 5. The five-minute tour for a new contributor

1. `npm run pipeline` — watch real USGS records show up in the log.
2. `cat public/data/manifest.json | jq` — see what got built.
3. Open `data/sites.json`. Each entry declares which connector stations inform it.
4. `npm run dev`, open http://localhost:3000. You should see ~10 colored pins on the inner-DMV bounding box.
5. Tap a pin. The detail card shows the grade, reason, signal breakdown, and a sparkline of historical grades (will be sparse until you've run the pipeline a few times).
6. Toggle "Paddle" / "Swim" in the bottom-left. Swim is disabled for DC-only sites with a tooltip.
7. Read [`GRADING.md`](./GRADING.md) and run `npm run grading:test` to see the rubric exercised against the 5 worked examples and 12 edge cases.

That's enough to start fixing bugs or extending features.

## 6. Adding a new data source

The connector contract is intentionally narrow: a TypeScript module exports `meta` and `fetch`, and emits `NormalizedRecord[]`. Adding a new one is one self-contained file plus one registration line.

### Step by step

```bash
# 1. Scaffold the directory
mkdir -p connectors/<source-id>/fixtures
touch connectors/<source-id>/{index.ts,README.md,index.test.ts}
```

### 2. Implement `index.ts`

Copy the shape of [`connectors/usgs-nwis/index.ts`](./connectors/usgs-nwis/index.ts) — it is intentionally the reference. The required exports:

```ts
import type { ConnectorContext, ConnectorMeta, NormalizedRecord } from '../shared/types';

export const meta: ConnectorMeta = {
  id: 'my-new-source',
  name: 'My New Source',
  cadence: 'hourly',
  license: 'public-domain',
  contact: 'https://example.gov/contact',
};

export async function fetch(ctx: ConnectorContext): Promise<NormalizedRecord[]> {
  // 1. Project ctx.sites into "stations my source informs"
  // 2. Make the HTTP call (use shared/http helpers)
  // 3. Map raw rows → NormalizedRecord shape (canonical units!)
  // 4. Return the array. Errors? throw ConnectorError; the pipeline degrades gracefully.
}
```

### 3. Register it

Add an import + entry to [`pipeline/connectors.ts`](./pipeline/connectors.ts):

```ts
import * as myNew from '../connectors/my-new-source';

export const ALL_CONNECTORS: Connector[] = [usgs, noaa, ark, doee, epa, myNew];
```

### 4. Test it

Write at least one fixture-driven normalization test mirroring [`connectors/usgs-nwis/index.test.ts`](./connectors/usgs-nwis/index.test.ts). Mock `globalThis.fetch` with a recorded response and assert on the emitted records.

### 5. Document it

Fill in `connectors/<source-id>/README.md` using the existing connector READMEs as a template. Add an entry to [`DATA_SOURCES.md`](./DATA_SOURCES.md) with the green-circle status icon.

### 6. Connect it to sites

If your source has stations that should inform particular recreation sites, add entries to `data/sites.json` under each affected site's `stations` array:

```json
{ "source_id": "my-new-source", "station_id": "STATION-123" }
```

Re-run `npm run validate:sites` to confirm the references are clean.

### Anti-patterns to avoid

These are documented in [`connectors/README.md`](./connectors/README.md) § Anti-patterns. The most important:

- Don't grade inside a connector. Connectors emit raw timestamped observations; the rubric in `grading/v1.ts` decides.
- Don't `console.log` — use the structured logger from `ctx.log`.
- Don't hardcode station IDs — they come from `ctx.sites`.
- Don't return partial data on error. Throw `ConnectorError`. The pipeline preserves the previous successful build's records for your source.

## 7. Curating sites

A "site" in this app is a *recreation* point, not a sampling station. [`docs/sites-curation.md`](./docs/sites-curation.md) is the canonical guide. Short version:

1. The location must be publicly accessible (no private docks).
2. People actually launch / swim / row from it.
3. It's within the inner-DMV bounding box (DC + Arlington + Alexandria + PG + Montgomery).
4. At least one of our data sources has a station within ~5 km.

To add a site, edit [`data/sites.json`](./data/sites.json), then `npm run validate:sites` to catch typos and `npm run pipeline` to regenerate the map.

## 8. Changing the grading rubric

Any change to thresholds, signal combination, or reason templates requires a new ADR per [`GRADING.md` § 8](./GRADING.md#8-versioning).

Workflow:

1. Open [`docs/adr/`](./docs/adr/) and copy `0004-grading-rubric-v1.md` to `0005-...`.
2. Document what you're changing and why (which data motivates it).
3. Update [`grading/v1.ts`](./grading/v1.ts).
4. Update or add tests in [`grading/v1.test.ts`](./grading/v1.test.ts) — the 5 worked examples in GRADING.md § 5 should still drive new tests if you change their expected outcome.
5. Run `npm run grading:test` and verify behavior under the change.

For larger changes — for example, adding a new signal type — ship the rubric as `grading/v2.ts` and keep `v1.ts` callable for historical re-grading.

## 9. Deployment

The system is designed to run statically on Cloudflare Pages + Cloudflare R2. The full topology is in [`ARCHITECTURE.md`](./ARCHITECTURE.md). The short version:

| Concern | Mechanism |
|---|---|
| Frontend hosting | Cloudflare Pages, auto-deploys on push to `main` |
| Data storage | Cloudflare R2 bucket (S3-compatible) |
| Scheduled connector runs | GitHub Actions cron (see `.github/workflows/connectors.yml`) |
| Frontend reads data from | `NEXT_PUBLIC_R2_BASE_URL` in production; `/data/` in dev |

### First-time deploy checklist

1. Create a Cloudflare account, link this repo to a Pages project.
2. Set Pages env vars:
   - `NEXT_PUBLIC_DATA_SOURCE=remote`
   - `NEXT_PUBLIC_R2_BASE_URL=https://<your-r2-public-url>`
   - `NEXT_PUBLIC_MAP_STYLE=osm-raster` (or `mapbox` if you have a token)
3. Create an R2 bucket (e.g. `dmv-water-watch-data`) and enable public access.
4. Configure CORS on the R2 bucket to allow `GET` from your Pages domain.
5. Set GitHub Actions secrets: `R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`.
6. Push to `main`. Cloudflare builds and deploys the frontend; the connectors workflow runs hourly and uploads artifacts.

If the R2 secrets aren't set, the connectors workflow still runs and uploads its artifact bundle as a GitHub Actions artifact (useful for debugging the pipeline).

### Switching to Mapbox tiles

1. Create a Mapbox account, generate a token, URL-restrict it to your production + preview domains.
2. Set `NEXT_PUBLIC_MAP_STYLE=mapbox` and `NEXT_PUBLIC_MAPBOX_TOKEN=<token>` in Pages env.
3. Redeploy. The map style switches without code changes.

## 10. Testing strategy

| Layer | Where | Cadence |
|---|---|---|
| Connector normalization | `connectors/<id>/index.test.ts` | Every PR |
| Grading rubric | `grading/v1.test.ts` | Every PR (and on every rubric change) |
| Site catalog | `npm run validate:sites` | Every PR + nightly |
| End-to-end pipeline | `npm run pipeline` in CI | Every PR + scheduled |
| Type safety | `npm run typecheck` | Every PR |
| Frontend build | `npm run build:next` | Every PR |

Manual smoke test before a release:

1. `npm run pipeline && npm run dev`
2. Verify map renders on mobile viewport (Chrome DevTools, iPhone simulator).
3. Tap a green pin, tap a yellow pin, tap an unknown pin — verify reason sentences make sense.
4. Toggle paddle/swim — verify pin colors update.
5. Open detail page directly via `/site/<id>` — verify it renders standalone.
6. Open methodology, about, sources pages — verify navigation works.

## 11. Coding conventions

- TypeScript strict mode. `noUncheckedIndexedAccess` is on; expect `T | undefined` everywhere you index.
- Default exports for React components; named exports for everything else.
- Tailwind for styles. CSS modules only if Tailwind is genuinely insufficient.
- Server components by default; `'use client'` only when you need browser APIs (MapLibre, `localStorage`, etc.).
- No `console.log` — use the structured logger in connectors / pipeline. Frontend can use `console.warn` for genuine warnings.
- No `any` unless interfacing with `navigator.share`-style browser APIs whose types are unstable; in that case, narrow at the boundary and explain in a comment.

## 12. Working with agents (Claude Code etc.)

The repo is structured to be agent-friendly:

- Every connector follows the same shape. An agent can read `usgs-nwis` and replicate the pattern.
- The grading function is pure with deterministic tests. Agents can refactor it safely.
- ADRs explain *why* we made each architectural choice, so agents don't fight the design.
- `npm run validate:sites` and `npm run typecheck` give fast, actionable feedback that agents can loop on.

If you're an agent reading this: start by reading `ARCHITECTURE.md` then `GRADING.md`. Then look at `connectors/usgs-nwis/index.ts` to understand the connector pattern. Most extensions are one connector + one site-list edit. Both can be PR'd independently.

## 13. Filing issues + PRs

- Bug reports: include the build ID (visible in the footer of `/sources`), browser + OS, and steps to reproduce.
- Feature requests: link to the relevant `REQUIREMENTS.md` or `ROADMAP.md` line. New features that aren't in the roadmap need a brief justification.
- New data source PRs: must include connector code + README + at least one test + entry in `DATA_SOURCES.md`. CI will refuse the merge otherwise.
